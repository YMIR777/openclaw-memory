# OpenClaw 记忆 / 搜索分层策略（2026-06-27）

## 目标

把当前 OpenClaw 从“所有能力混在默认路径里”调整为：

1. 默认路径轻、快、稳定。
2. 深搜路径显式触发，不拖慢日常回复。
3. Wiki 层负责可追溯知识沉淀，不抢主 recall。
4. Session transcript 只在明确需要找旧聊天时进入搜索面。
5. 主动记忆能力先小范围评估，不直接全局开启。

## 当前配置结论

### 已采用

- `memory.backend = "qmd"`
- `memory.qmd.searchMode = "query"`
- `memory.qmd.rerank = false`
- `agents.defaults.memorySearch.sources = ["memory"]`
- `agents.defaults.memorySearch.experimental.sessionMemory = false`
- `memory-wiki` 已启用 bridge + Obsidian render
- `memory-core.dreaming.enabled = true`，且 storage 为 `separate`
- `active-memory` 已关闭（等待后续单独重调）
- 模型 fallback 已跨上游：`gpt/gpt-5.4` → `gpt/gpt-5.5` → `lumina/gpt-5.5` → `wintoken/deepseek-v4-pro`

### 实测证据

- QMD 已从 `2.0.1` 升级到 `2.5.3`。
- OpenClaw 管理的 QMD index 位于：
  - `/Users/Ymir/.openclaw/agents/main/qmd/xdg-cache/qmd/index.sqlite`
- 该 index 当前状态：
  - 709 files indexed
  - 2583 embedded
  - sources = memory
- 默认 `qmd search` 对当前问题约 0.27 秒返回。
- QMD 2.0.1 下，隐式 `qmd query "..."` 会长时间停在 `Expanding query...`。
- 现已直接补丁 OpenClaw 内置 QMD manager：当 `searchMode=query` 且 `rerank=false` 时，内置 `memory_search` 不再走 plain query，而是走 typed `lex+vec` 查询文档，并跳过 rerank。
- 因此默认主路现在重新升级为 `searchMode=query + rerank=false`，同时保留显式 deep-search 工具做更大候选池/更长超时的语义召回。

## 四层搜索策略

### Layer 1：默认增强搜索（日常主路）

用途：普通 memory recall、近期系统记录、明确关键词、文件路径、项目名、人物名，也覆盖一部分语义相似召回。

实现：

- 使用 OpenClaw 内置 `memory_search`。
- 配置保持：
  - `memory.qmd.searchMode = "query"`
  - `memory.qmd.rerank = false`
  - `sources = ["memory"]`
  - `timeoutMs = 8000`

原因：

- 现在内置调用已经走 typed `lex+vec` 快路径，不再是旧的 plain query。
- 比纯 BM25 更有上限：包含 lex + vec 混合召回。
- 跳过本地 reranker，避免最慢的 QMD LLM rerank 链路。

### Layer 2：显式深搜工具（可控语义召回路径）

用途：默认 recall 不足时，手动扩大候选规模或显式启用语义混合召回。

实现：

- 使用本地工具：`tools/memory-deep-search`
- 该工具运行在 OpenClaw 管理的 QMD XDG 环境中，避免误用裸 QMD 默认 index。
- 查询方式使用 QMD 2.5.3 typed query：`lex + vec`，并传 `--no-rerank`。
- 工具内置超时保护：默认 20 秒，超时后退出，不阻塞主工作流。
- 默认参数：
  - `--limit 8`
  - `--candidate-limit 12`
  - collections: `memory-root-main`, `memory-dir-main`

命令示例：

```bash
/Users/Ymir/.openclaw/workspace/tools/memory-deep-search "OpenClaw QMD searchMode sessions" --timeout 20
```

原则：

- 默认主路先走内置 `memory_search`。
- 当需要更大候选池、更完整 JSON 或显式语义召回时，再用这个工具。
- 不使用 QMD 默认 rerank，除非未来单独证明 rerank 速度和收益都合格。

### Layer 3：Wiki 知识层

用途：稳定实体、长期项目、系统接管记录、可追溯结论、矛盾与证据。

实现：

- 保持 `memory-wiki` bridge 模式。
- 用 `memory_search corpus=all` 做广义召回。
- 需要 provenance / page-level 结构时，用 wiki 原生工具（如可用）：`wiki_search` / `wiki_get` / `wiki_apply`。

原则：

- Wiki 不替代 active memory / QMD。
- Wiki 负责“整理后的知识”，QMD 负责“原始笔记可搜”。
- 接管记录、模型路由、系统经验应优先沉淀成 wiki/entity/concept 页面。

### Layer 4：Session transcript 深搜（暂不默认开启）

用途：用户明确要找“某次旧聊天里说过什么”。

当前策略：

- `agents.defaults.memorySearch.sources = ["memory"]`
- `memory.qmd.sessions.enabled` 暂不启用。
- `sessionMemory = false`

原因：

- session transcript 面积大、噪音多、容易拖慢默认 recall。
- 当前主要问题是系统稳定性，不是最大化历史聊天召回。

未来触发条件：

- 用户明确频繁需要“从旧聊天找原话”。
- QMD typed deep search 稳定。
- session export retention 有明确边界，例如 14-30 天。

## Active Memory 建议

官方 active memory 是“主回复前的阻塞记忆子代理”。它很适合小黒这种强连续上下文场景，但不能无差别全局开启。

当前已关闭，原因是试运行期间只有 `timeout`、没有稳定摘要收益，并且一度疑似干扰主 memory manager。

后续重开条件：

- 默认 `memory_search` 主路连续多轮稳定。
- 先用 `gpt/gpt-5.4` + `thinking=off` 单独小范围回归。
- 若重开，继续固定 `qmd.searchMode = search`，不与主路 `query` 争抢混合检索链路。

设计原则：

1. 只在主 Agent 的 direct 会话运行，不进 group/channel。
2. 只看最近上下文，不扫全历史。
3. 不保存 active-memory 子代理 transcript，避免新增会话垃圾。
4. 用 3.5 秒超时 + 熔断保护，避免它长期拖慢主回复。
5. Active Memory 单独固定走 `qmd.searchMode = search`，不继承主路 `query`，避免它和主 recall 争抢较重的混合检索链路。
6. 如果仍出现 `failed` / `timeout` 或影响主路 memory manager，优先关闭 Active Memory，而不是继续加预算。

## 操作规范

### 默认回答前

需要 prior work / 决策 / 偏好 / todo 时：

1. 先 `memory_search corpus=all` 或默认 memory recall。
2. 若结果不足，再用 `tools/memory-deep-search`。
3. 若需要证据结构，再查 wiki。
4. 若用户明确说“旧聊天里”，再考虑 session history / transcript 搜索。

### 什么时候用 deep search

- 用户说“我之前是不是说过……”但关键词不确定。
- 需要跨多个主题找共同模式。
- 默认 memory_search 命中少或明显偏。
- 需要找概念相似而不是词面相同的内容。

### 什么时候不用 deep search

- 查当前配置。
- 查明确文件路径。
- 查最近接管记录。
- 普通技术解释。
- 默认 memory_search 已经高质量命中。
