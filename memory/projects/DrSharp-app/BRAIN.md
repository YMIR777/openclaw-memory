## 一、这个项目是怎么来的

### 触发点
2026-05-25，小黒完成了 Dr. Sharp 分析 #010。在这次分析中，他描述了最近的困境（作息好了但空虚还在、变菜了不想接陪玩、新认识了潜江女生、感觉"生活要踏上正轨了"但仍然没有产出）。

分析结束后，小黒说他有一个想法：
> "做一个应用，用来记录我们每次和 Dr. Sharp 的对话。我们从最开始就做好了数据存储，每次的对话数据都有。做一个美观、可视化、并且可以实时提醒我我需要注意什么。这些经验和总结才是真正有价值的，我需要展示出来。我可以不再被短视频吸引注意力，所以这个应用就是让我能查看我真正需要的是什么，也能很好的提醒我。"

### 用户的核心需求
这个应用不只是展示历史分析——它是小黒的**成长可视化系统**。他做了两年 aim training 但没有真正的创作产出。这个应用是他的"第一格"——第一个为自己创造的东西。

小黒在开发中的深层需求（他自己说的）：
> "我不需要知道 shader 具体怎么写，但是我至少需要知道它的存在、它的价值并且如何使用。在 vibe coding 的过程中如果我不知道这些内容，我很容易像井底之蛙一样困在狭小的认知里。所以我需要你在开发的过程同时帮我认识这些我以前认知之外的内容。"

> "在开发的过程中如果我自己不清楚到底怎么实现，我就很难精细调整和实现我想要的效果，因为文字的表达和理解是会有歧义和偏差的。"

> "但我知道具体需要什么库也许我才真的有确认感。"

## 二、技术选型背后的决策过程

### 为什么是 R3F（React Three Fiber）

小黒一开始就知道 Three.js，但他需要的是"组件化地使用 3D"，不是手写原生 Three.js。他之前在"大金库"项目里接触过 R3F，但是"什么都不懂就直接用"。

小黒的原话：
> "我也想要 R3F，我能接受学 R3F！之前我只是什么都不懂就用，当你如果要帮我学习了解，这太好了，比我什么都不懂直接做更好，并且我还能学到知识。"

**决策过程：**
- 纯 CSS 3D transforms — 环形布局和深度效果有限
- 原生 Three.js — 代码量大、没有组件化
- R3F — React 生态、组件化、`@react-spring/three` 可以做弹簧物理动画、`@react-three/drei` 和 `@react-three/postprocessing` 生态丰富
- **选了 R3F**

### 为什么是 preText

小黒是在 preText 的演示网站（pretextjs.net/zh）上看到效果后主动提出的：
> "这些文字效果和可以实现的内容太酷了，我必须要在我的这个应用里面使用！"

preText 的核心价值：
- 纯数学计算文本高度/布局，不触发浏览器 layout reflow
- 比 DOM 测量快 300-600 倍
- 支持 Canvas/SVG/WebGL 渲染
- 有 Rich Text 内联元素支持（标签徽章可以直接嵌入文本流）
- 作者是 Cheng Lou（前 React 核心团队，现 Midjourney 前端）
- 5KB gzip

**与项目的关系：**
- 每个 Dr. Sharp 分析都是一篇长文（4000-7000 字）
- 需要在波浪形边框内精确排版
- 语音播放时需要精确知道每个段落的 y 坐标来滚动
- 标签需要在文本里以内联徽章形式显示——preText 的 rich-inline 直接支持

### 为什么不用 Superpowers skill

Superpowers 是子 agent 驱动的并行开发流程（Brainstorm → Plan → Subagent-Driven Build → Code Review）。对于小黒的项目不合适，原因：
- 这是小黒要亲手学 R3F 的项目，不是甩给 agent 跑的项目
- 他和我在同一个会话里一步步走，不需要 spawn 子 agent
- Karpathy 四原则已经覆盖了 Superpowers 的核心（Think → Simplify → Surgical → Goal-Driven）

### 为什么用 design-taste-frontend（但挑着用）

这个 skill 有两半：
- **工程纪律**（全用）：不用 h-screen 用 dvh、动画只动 transform/opacity、效果层 pointer-events-none、全覆盖交互状态。这些防止常见的性能/兼容性坑。
- **审美规则**（挑用）：禁纯黑 → 豁免（深空黑是设计基础，而且用的是 `#0a0a0f` 不是真纯黑）、禁外发光 → 部分豁免（标签粒子微弱光晕是 3D 空间感需要）

小黒问过这个 skill 会不会限制发挥：
> "你觉得 taste skills 有用吗，它是在固化限制你的发挥，还是真的可以帮助你提高质量呢？"

Dr. Sharp 的回答：设计 skill 没有限制发挥——它提供了一个工程质量的底线（告诉你哪些坑不用踩），而小黒的想象力在它之上自由发挥。

### 为什么不用 elite-frontend-design

与 design-taste-frontend 大量重叠但工程纪律更少。两套规则会打架（一个说用 Geist，一个说用 JetBrains Mono）。选一套就够了。选了 design-taste-frontend 因为它有更多工程保护规则。

## 三、主页面——完整设计细节

### 小黒的手绘草稿
文件位置：`主页面-草图.jpg`（与本文档同目录）

小黒画了一个环形卡片轮播的草稿。他特意说明：
> "我说的左右箭头是我画的概念图里面方便你理解才画的，并不是需要左右箭头滚动，而是自己左右拖动就可以滑动，并且滑动效果要丝滑，并且带有加速度，比如果我速度很快并且滑动范围很大的左右滑动卡片的时候，卡片要全部一起转圈转很久才停下来，这种带有自然的物理效果的交互的感觉特别好。"

**3D 环形卡片轮播：**
- 笔记围成一圈，在深黑 3D 空间中排列
- 交互：手指拖动 → 环形旋转 → 松手 → 惯性衰减（角速度 × 摩擦系数 0.95/帧）→ 吸附最近卡片
- 静止态：极缓慢自转（几乎察觉不到）+ 每张卡片微微上下浮动（sin 波，不同卡片相位不同）——"呼吸感"
- 多卡片：自适应间距堆叠
- 少卡片：宽松间距

**其他元素：**
- 左上角搜索框：全局搜索所有笔记内容
- 底部圆形语音按钮：琥珀色/暖橙色，微弱脉冲光晕（心跳感），视觉锚点
- 标签云 3D 粒子：环形圆心悬浮，发光 + 拖尾，选中筛卡片
- "每日一句"：环形中心显示核心洞察文字，淡入，停留 5 秒

**交互细节：**
- 长按卡片 → 浮出放大预览 → 松手弹簧弹回（像从书架抽书看封面）
- 点击卡片 → 水波纹过渡进入详情页
- 标签选择 → 相关卡片亮起、无关变暗

**色彩理念：**
深黑不是纯黑。用 `#0a0a0f` 左右的深蓝灰，叠加极微弱的远处光粒子——"像在太空里看到遥远星光"。3D 物体在这种背景上才有深度感。纯黑会吃掉所有阴影和景深。

小黒对 Dr. Sharp 提出的视觉建议全部说了"采纳"——包括呼吸感、标签粒子化、深空背景、长按预览、语音按钮心跳光晕。

## 四、笔记详情页——完整设计细节

### 小黒的手绘草稿
文件位置：`详情页-草图.jpg`（与本文档同目录）

小黒的描述：
> "从主页面的卡片里面进行动画，卡片变大，边界开始模糊，像水一样的波纹浮现，背景虚化。然后这里我们可以用 pretext 设计很多有意思的效果。笔记边界呈现如图所示，在笔记边界内上下滚动可以调整进度，并且配合上语音输出，要实现自动滚动的效果。"

**过渡动画流程：**
1. 卡片从小变大（scale up）
2. 边缘开始模糊（shader noise displacement）
3. 水波纹从中心向外扩散（ripple shader）
4. 背景虚化（depth of field / backdrop blur）
5. **最终形态：巨大的、不规则的波浪形边框**（不是矩形！）

**内容区：**
- preText 精确排版分析全文
- 波浪形边框内上下滚动
- 标签内联徽章显示（preText rich-inline）
- TTS 语音播放（V2）：朗读分析，段落级绑定，点击段落播放
- 语音播放时自动滚动跟随（preText 提供精确 y 坐标，不需要 DOM 测量）
- 标签关联推荐：边缘浮现关联标签

## 六、认知扩展——小黒需要知道"存在什么"

### Shader / GLSL（已介绍）
在 GPU 上跑的小程序。做视觉特效——水波纹、噪声位移、发光、景深。R3F 生态里用 `@react-three/postprocessing` 和 `@react-three/drei` 可以不用手写原生 GLSL。

### Post-processing / Bloom（已介绍）
渲染完场景后叠加的效果层。Bloom（发光物体的光晕扩散）、Vignette（边缘变暗）、Depth of Field（景深模糊）。

### SVG Filter（已介绍）
`feTurbulence` + `feDisplacementMap` 可以给任意元素加有机波动效果。波浪形边框的 2D 实现方式之一。

### 粒子系统（已介绍）
Points + BufferGeometry 实现大规模粒子效果。R3F 的 drei 有 `<Sparkles>` 组件。用于深空星光、标签拖尾、卡片周围微尘。

### 字体作为图形元素（已介绍）
大号中文标题溢出卡片边缘——不是标题，是图形。preText 精确测量保证不溢出。

## 九、开发流程与原则

### Karpathy 四原则（已确认使用）
1. Think Before Coding — 我们已经做了（草图 + 本文档）
2. Simplicity First — MVP 只做展示器，V2 预留接口
3. Surgical Changes — 每次只推进一个功能
4. Goal-Driven Execution — 每步有可验证的结果

### design-taste-frontend 关键工程规则
- `min-h-[100dvh]` 不用 `h-screen`
- 动画只动 `transform` / `opacity`
- 效果层加 `pointer-events-none`
- 全覆盖交互状态（loading/empty/error）
- 字体：Geist / Satoshi / JetBrains Mono（不用 Inter）

### 审美豁免
- 不用纯黑的规则 → 豁免（深空黑是设计，不是偷懒，而且用的不是真纯黑）
- 不用外发光的规则 → 部分豁免（标签粒子微弱光晕是 3D 空间感需要）

## 十二、Pivot 记录 — 为什么从 3D 切到 2D

> 日期：2026-06-11 · 决策：小黒 + 黒

按 SPEC v1 搭了完整 R3F 骨架，但反复出问题——missing import 黑屏三轮、postprocessing HMR 不生效、drag 和 animate 冲突。

小黒主动提出切换："做 3D 场景对你目前 DeepSeek V4 的模型能力来说比较复杂，交付很差需要反复修改耗时耗力。"

**理由：** R3F 调试成本太高、Dr. Sharp 核心价值是"分析文字以有尊严有温度的方式呈现"而非 3D、design-taste-frontend 为 2D 设计、Framer Motion 迭代更快。

**新架构：** Framer Motion 替代 R3F、CSS scroll-snap 替代 3D Ring、DOM 段落替代 Canvas 3D 文字。preText 保留（排版计算用，渲染用 DOM）。

**保留：** 暖色调配色、Geist+Georgia 字体、design-taste 纪律、Karpathy 原则、数据模型。

**Bundle：** 1362 KB → 383 KB（砍掉近 1MB）

## 十三、2D 开发关键决策

### CSS scroll-snap 替代 Framer Motion drag（2026-06-12）
Framer Motion `drag` + `animate` + `constraints` 持续冲突：卡片只显示一张、拖拽松手误触点击。CSS scroll-snap 让浏览器自己区分滚/点，零 JS 吸附。焦点公式：`scrollLeft / STEP`。

### 水波纹 → 段落悬停微高亮（2026-06-11）
preText Canvas 版和 DOM 逐字 span 版均存在可读性问题。小黒："不是最适合我们项目的"。改为段落悬停微高亮——Dr. Sharp 核心是深读，非花哨动效。
