# 2026-04-10 日志

## 重大进展：macOS 自动化权限问题彻底解决

### 问题描述
- OpenClaw cron 定时任务执行 osascript 时失败（需要审批）
- osascript 控制网易云音乐时 TCC 权限不足
- 每次新 Terminal 环境需要手动一个一个点授权

### 根因分析
1. OpenClaw exec-approvals 白名单里 osascript 条目是空对象 `{}`，被当作"未配置"忽略
2. macOS TCC 权限按 App Bundle ID 缓存，不是按进程路径
3. osascript 的 bundle ID = `com.apple.osascript`，所有会话共享同一个 TCC 权限上下文
4. cron/isolated 环境下 TCC 权限和用户桌面环境是相同的

### 已解决的问题

**1. osascript exec-approvals 白名单**
- 执行 `openclaw approvals allowlist add --agent main "/usr/bin/osascript"`
- osascript 已加入白名单，exec 层不再拦截

**2. TCC Automation 权限**
- 验证结果：osascript 在 cron 环境里**完全可用**
- cron 环境测试（16:01）：网易云音乐暂停成功 ✅
- 原因：macOS TCC 权限按 Bundle ID 缓存，osascript 权限可被所有会话继承

### 重要教训
1. **不要猜测权限状态** — 用实际命令测试验证，不只看系统设置里的显示
2. **osascript 是全局权限** — osascript 对某 App 的授权，所有通过 osascript 访问该 App 的进程都共享
3. **cron 环境不是独立的 TCC 上下文** — 它和用户桌面共享同一个 osascript 权限缓存
4. **Google Chrome 例外** — Chrome 有自己独立的权限机制，需要单独授权

### 批量授权脚本
路径：`~/.openclaw/scripts/tcc-batch-auth.sh`

下次新 Terminal 环境时运行此脚本，一次性触发所有常用 App 的授权对话框：
```bash
bash ~/.openclaw/scripts/tcc-batch-auth.sh
```
