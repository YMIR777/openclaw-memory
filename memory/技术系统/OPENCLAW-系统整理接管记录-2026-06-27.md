4. 给出最小改动方案，把 “memory search 经常报错” 先压下去。

### 本阶段已得到的根因判断
- 当前 `memory.backend = "qmd"`，所以真实 `memory_search` 走的是 QMD sidecar，而不是 builtin SQLite memory engine。
- 当前配置里 `memory.qmd.searchMode = "query"`，同时 `memory.qmd.limits.timeoutMs = 10000`。
- 源码与实测都对上了这一点：OpenClaw 会直接调用 `qmd query ...`，并把 `timeoutMs` 作为该搜索流程的超时预算。
- QMD 官方文档明确说明：
  - `query` 模式包含 query expansion / reranking / vector 路径
  - 首次或未完全预热时会明显变慢
  - 若慢机器或本地 sidecar 较重，建议提高 timeout，或改用 `search` 模式
- 本机实测：
  - `qmd search "openclaw memory search timeout" --json -n 20 -c openclaw-memory` 约 `0.33s` 完成
  - `qmd query "openclaw memory search timeout" --json -n 20 -c openclaw-memory` 卡在 `Expanding query...`，随后被杀
- 当前 `qmd status` 同时显示：
  - `465 files indexed`
  - `896 embedded`
  - `293 pending embeddings`
  - `Embedding` / `Reranking` / `Generation` 模型均启用
- 结合以上证据，当前超时主因不是索引损坏，而是：
  - **QMD `query` 模式本身过重**
  - **10 秒超时太短**
  - **293 pending embeddings 让语义路径更不稳定**

### 已落地的最小修法
- 已把 `memory.qmd.searchMode` 从 `search` 升级为 `query`，并设置 `memory.qmd.rerank=false`。
- 已把 `memory.qmd.limits.timeoutMs` 调整为 `8000`。
- 已把 `agents.defaults.memorySearch.sources` 从 `["memory", "sessions"]` 简化为 `["memory"]`。
- 已把 `agents.defaults.memorySearch.experimental.sessionMemory` 设为 `false`。
- 已把模型 fallback 改成跨上游：
  - primary: `gpt/gpt-5.4`
  - fallbacks: `gpt/gpt-5.5` → `lumina/gpt-5.5` → `wintoken/deepseek-v4-pro`
- 已补齐 OpenClaw 管理 QMD index 的 pending embeddings：该 index 当前约 `705 files indexed / 4384 embedded / 0 pending`。
- 已新增分层搜索策略文档：`memory/topics/OPENCLAW-记忆搜索分层策略-2026-06-27.md`。
- 已新增实验性语义深搜工具：`tools/memory-deep-search`。
- 该工具使用 OpenClaw 管理的 QMD XDG 环境，避免误用裸 `~/.cache/qmd` 默认 index。
- 该工具使用 QMD 2.5.3 typed `lex+vec --no-rerank`，并带超时保护；实测技术/中文查询约 1.0-1.2 秒返回。
- 已升级 `@tobilu/qmd`：`2.0.1` → `2.5.3`。升级前已备份旧包与 OpenClaw QMD index；升级后通过 `qmd cleanup` 修复旧 index 的 CJK FTS migration constraint。
- 已开启 Active Memory 小范围试运行，但实测一度连续 `failed/timeout`。现已进一步收紧：只限 `main` + `direct`，固定 `model=gpt/gpt-5.4`、`thinking=off`、`queryMode=message`、`promptStyle=precision-heavy`、`timeoutMs=3500`、`qmd.searchMode=search`。目标是把它从“可能干扰主路”的阻塞记忆子代理，降成“低预算的轻量前置 recall”。
- 这样做的目的不是追求最强语义召回，而是先把“memory search 经常报错”压下去，并降低单一上游故障导致主模型与 fallback 一起失效的风险。
- `search` 是 BM25-only 路径，速度明显更快，而且 docs 明确说明该模式会跳过 QMD 的向量就绪探测与 embedding 维护。

### 对 293 pending embeddings 的判断
- 当前不建议把它当成第一优先级故障去硬冲完成。
- 原因：
  - 现在最痛的是查询超时，不是索引不可用
  - 切到 `search` 后，pending embeddings 不再阻塞正常 recall
  - 若后面想恢复高质量语义搜索，再单独安排 `qmd embed` 补齐更合理
- 结论：
  - **短期先不强制 embed 到完成**
  - **先观察切到 `search` 后 memory search 是否稳定**

### 配置职责边界（当前阶段结论）
- `memory.backend = qmd`
  - 决定 OpenClaw 真实使用哪个 memory engine
- `memory.qmd.*`
  - 决定 QMD 的搜索模式、超时、paths、scope、sessions、update/embed 行为
- `agents.defaults.memorySearch.*`
  - 是通用 memory search 配置层；对 builtin / provider / embedding 语义更关键
  - 但在 `backend = qmd` 时，真正的 QMD 查询性能主要受 `memory.qmd.*` 支配，而不是这里的 `provider/model`
  - 其中 `sources` 很关键：
    - `["memory"]` = 只搜 `MEMORY.md` + `memory/*.md`
    - `["memory", "sessions"]` = 额外把历史聊天 transcript 也纳入 recall 面
- `plugins.entries.memory-core`
  - 主要负责 active memory / dreaming / recall orchestration
- `plugins.entries.memory-wiki`
  - 负责 wiki 编译与 bridge，不是这次查询超时的主因
