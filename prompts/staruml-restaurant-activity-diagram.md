# Claude Code 提示词：StarUML 餐馆订餐系统"记录预约"活动图

## 任务目标

在 StarUML 中绘制一个**带泳道的活动图**，完成"餐馆订餐系统"的"记录预约"用例建模。

## 操作步骤（严格按照顺序执行）

### 第 1 步：打开 StarUML
```bash
open -a StarUML
```
等待 3 秒，让应用完全启动。

### 第 2 步：新建模型
1. 点击菜单 `File` → `New`（或快捷键 `Cmd+N`）
2. 创建新项目后，右键左侧边栏 → `Add Diagram` → `Activity Diagram`
3. 把 diagram 改名为"记录预约活动图"

### 第 3 步：创建泳道（3 个泳道）

按以下顺序添加泳道（泳道名就叫这个，不用翻译）：

- **Customer**（顾客）
- **Restaurant**（餐厅）
- **System**（系统）

添加方式：在活动图空白处右键 → `Add Swimlane` → 输入名称。  
三个泳道从左到右排列。

### 第 4 步：按以下内容绘制活动图

**Customer 泳道：**
1. 初始节点（实心圆）→ 标签："开始预约"
2. 动作节点："输入预约信息（姓名、电话、人数、日期、时间）"
3. 动作节点："提交预约请求"

**System 泳道：**
4. 动作节点："接收预约请求"
5. 动作节点："验证预约信息（检查必填项、日期格式）"
6. 菱形判断节点："信息有效？"
   - 是 → 继续向下（System 泳道）
   - 否 → 流向 Customer 泳道"返回修改请求"
7. 动作节点："检查餐桌可用性"
8. 菱形判断节点："有可用餐桌？"
   - 是 → 继续
   - 否 → 流向 Customer 泳道"提示无可用桌位"
9. 动作节点："创建预约记录"
10. 动作节点："发送预约确认通知"

**Restaurant 泳道：**
11. 动作节点："处理预约确认"
12. 活动节点（粗边框）："完成预约登记"
13. 结束节点（实心圆+外圈）

### 第 5 步：连接线和标注

- 每个判断节点的两个出口分别标注"是/Yes"和"否/No"
- 从"返回修改请求"连接回 Customer 泳道的"输入预约信息"，形成循环（注意不要画成死循环，标注最多循环 3 次）
- 从"提示无可用桌位"连接到 Customer 泳道的"提交预约请求"前（重新选择时间）

### 第 6 步：美化
- 字体大小统一 12pt
- 动作节点圆角矩形
- 各泳道用不同颜色区分（Customer：浅蓝，Restaurant：浅绿，System：浅黄）

## 注意事项

⚠️ **重要**：
1. 每完成一个泳道的节点后，描述一下当前状态，**不要一次性自动执行所有操作**
2. 如果 StarUML 弹出任何对话框，先停止并报告给我
